# Инженерный стиль кода IWE

> Для разработчиков, пишущих или правящих код в репозиториях IWE. Ссылка из [developer-guide.md](developer-guide.md) на станции 4 (Работа).

## Принцип

База стиля не описывает «хороший код» — это список конкретных запахов с парой «было → стало». **Вкус — это отсутствие запахов.**

**Детектор контекста:** «вернётся ли кто-нибудь к этому коду?» Одноразовый скрипт/прототип — не плоди абстракции. Боевой код (main, shared-модуль) — все правила обязательны.

**Граница с жёсткими запретами:** `git add -A`, секреты в логах, `--no-verify` — это правила безопасности агента (PACK-agent-rules), не сюда. Тест: «ломает систему/безопасность?» → там. «Делает код хуже на глаз профессионала?» → здесь.

## Правила (P0–P12)

| Правило | Было → стало (кратко) |
|---------|------------------------|
| **P0** — форматтер и линтер перед коммитом | Нет конфига → спроси, не выдумывай стиль |
| **P1** — тест проверяет наблюдаемый результат | `assert True` — запах; нужен assert конкретного эффекта |
| **P2** — третье повторение → функция | Не `locals()[str]` — реальные объекты в словаре/списке |
| **P3** — удалить мёртвую ветку | Не оставлять «для совместимости» — вернётся из истории при нужде |
| **P4** — `except: pass` без лога запрещён | Голое исключение без типа и действия — всегда запах |
| **P5** — длинную функцию со смешанными обязанностями разбить | Контр-критерий (Ousterhout): не дроби глубокий модуль ради длины — дроби по смене уровня абстракции |
| **P6** — структурированный лог на входе и при отказе | Обработчик без сигнала на входе/падении — запах |
| **P7** — ASCII-баннер внутри функции = скрытая декомпозиция | Секция под баннером → отдельная функция |
| **P8** — идиома вместо борьбы с языком | `range(len(x))` вместо `for item in x`; идиомы по языкам — ниже |
| **P9** — библиотека вместо ручного парсинга стандартного формата | CSV/JSON/дата руками → `csv`/`json`/`date.fromisoformat` |
| **P12** — абзац-оправдание костыля сигналит незакрытую первопричину | Длинное объяснение «почему нельзя трогать» → фикс первопричины или однострочный WHY-якорь на тикет |

## Примеры было/стало

**P1 — наблюдаемый результат:**
```python
# было:
await dp.feed_update(bot, update); assert True
# стало:
await dp.feed_update(bot, update); assert len(bot.sent) > 0
```

**P4 — лог вместо проглоченного исключения:**
```python
# было:
try: await conn.close()
except Exception: pass
# стало:
try: await conn.close()
except Exception: log.warning("conn.close failed", exc_info=True)
```

**P8 — идиома:**
```python
# было:
result = []
for x in items:
    if x.active: result.append(x.name)
# стало:
result = [x.name for x in items if x.active]
```

**P12 — WHY-якорь вместо абзаца-оправдания:**
```python
# было: абзац, оправдывающий костыль (история инцидента, «не трогать без ревью»)
conn.close()
# стало:
conn.close()  # WHY: драйвер X не освобождает fd на таймауте (INC-4821, driver v2.3)
```

Полные примеры и идиомы по языкам (Python/TypeScript/Go/Rust/Bash) — в `PACK-digital-platform/pack/digital-platform/02-domain-entities/engineering-code-style-base.md` (источник истины, доставляется в контекст агента автоматически хуком перед правкой кода).

## Работа с Claude

- Правила P0-P12 впрыскиваются в контекст агента автоматически перед правкой кода (хук `inject-code-style.sh`) — не нужно напоминать вручную.
- Если правка выглядит как костыль (нужен абзац-оправдание) — спроси Claude про первопричину до того, как принимать код.
- Ревью PR: смотри на знакомые запахи из таблицы выше — расхождение = комментарий со ссылкой на номер правила (например «P5 — раздели на 3 функции»).
