# Работа с Claude в разработке

> Для разработчиков, использующих Claude Code при написании/правке кода IWE. Ссылка из [developer-guide.md](developer-guide.md) на станции 4 (Работа) — рядом с [тестированием как спецификацией](testing-as-spec.md) и [инженерным стилем кода](code-style.md), которые Claude обязан соблюдать в вашей сессии не меньше, чем вы сами.

## Когда Claude полезен, когда опасен

**Полезен:**
- Механическая работа с известным контрактом — миграция N файлов по одному паттерну, рефакторинг под уже согласованный интерфейс, написание тестов границы (см. [testing-as-spec.md](testing-as-spec.md)).
- Разведка кодовой базы перед правкой — «где это уже реализовано», «что вызывает эту функцию».
- Черновик там, где вы сами знаете, как проверить результат (closed-loop задача — есть тест/чеклист).

**Опасен:**
- Задачи, где сама постановка нечёткая (problem-framing/open-loop) — Claude с готовностью предложит решение для неверно понятой задачи, не спросив.
- Код на границе систем без явного контракта — без спецификации Claude угадывает границу, а не читает её у вас в голове.
- Всё, что вы не можете быстро проверить сами — если ревью решения занимает у вас больше времени, чем самостоятельное написание, делегирование не окупилось.

## Prompt-паттерны: «объясни» vs «сразу пиши»

**«Объясни как будешь делать, потом пиши»** — для нетривиального: новая функция с побочными эффектами, правка на границе модулей, что угодно, где неверное направление дорого обнаружить постфактум. Просите план в 3-5 пунктов до кода. Если план ошибочен — дешевле поймать до написания 200 строк, чем после.

**«Сразу пиши»** — для тривиального и обратимого: однострочный фикс, тест по уже описанной спецификации, переименование по явному списку. Промежуточное объяснение здесь — трата времени обеих сторон.

**Как распознать, что взяли не тот режим:** если после «сразу пиши» вы тратите больше времени на понимание диффа, чем заняло бы объяснение заранее — в следующий раз берите план сначала.

## Стоп-сигналы: когда Claude пишет мёртвый код

- **Третье повторение вместо функции (P2, [code-style.md](code-style.md)).** Claude склонен копировать похожий блок третий раз вместо того, чтобы предложить общую функцию — попросите явно: «вынеси общую логику», если видите третье почти-дублирование.
- **Мёртвая ветка «для совместимости» (P3).** Просьба «не удаляй старый путь, вдруг понадобится» без реального потребителя — тот самый запах P3. Если ветку никто не вызывает — она должна уйти, а не остаться меченой TODO.
- **Абстракция под гипотетическое будущее.** Флаг режима, параметр «на будущее», интерфейс под второго потребителя, которого ещё нет — Claude охотно проектирует расширяемость, которую никто не просил. Спрашивайте: «а кто сегодня это использует?»
- **Обработка ошибок для сценария, который не может произойти.** `try/except` вокруг внутреннего вызова, который не может упасть по контракту — лишний код, который прячет реальные баги под тихим `pass`.

## Чеклист перед merge: что проверить вручную, что оставить Claude

**Оставить Claude (он делает это надёжно и это дёшево перепроверить):**
- Форматирование, линтер (P0) — механическая проверка.
- Соответствие уже написанной спецификации теста (border-контракт из testing-as-spec.md).
- Список изменённых файлов и краткое summary диффа.

**Проверить вручную, не доверять отчёту Claude «сделано и работает»:**
- **Границу системы** — Claude может написать тест, который проходит, но проверяет не то, что нужно (см. стоп-сигнал в testing-as-spec.md: тест ломается при равноценной замене реализации → тестирует деталь, не контракт).
- **Побочные эффекты вне анонсированного списка файлов** — `git diff --stat` целиком, не только файлы, которые обсуждали.
- **Секреты и credentials в диффе** — Claude иногда логирует значения переменных окружения при отладке; grep на паттерны токенов перед коммитом.
- **Реальный сценарий отказа** — «что будет, если сеть недоступна / вход пустой / файл не существует» — Claude по умолчанию пишет happy path, если явно не попросить режим отказа.
- **Двойной выход** (см. [developer-guide.md](developer-guide.md) станция 4) — код без зафиксированного знания (различение/memory/Pack) считается незакрытым, Claude сам не остановит вас на этом шаге, если не спросить прямо «что здесь стоит зафиксировать».
